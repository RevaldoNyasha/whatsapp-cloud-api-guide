Webhooks

Before you write a single line of backend code, you need to understand what a webhook is and how Meta uses it to deliver messages to your server. This page covers the concept, the configuration, and every setting you need to get right.

What is a webhook?

A webhook is a way for one server to notify another server that something happened — in real time, automatically, without polling.

Without webhooks, to check if a new WhatsApp message arrived, your server would have to keep asking Meta's API every few seconds: "Any new messages? Any new messages?" — that's called polling, and it's slow, wasteful, and impractical at scale.

With a webhook, Meta does the opposite. Meta calls your server. The moment a user sends a WhatsApp message to your number, Meta's system immediately fires an HTTP POST request to a URL you specify — your webhook URL. Your server receives that POST, reads the message data, and handles it.

Without webhooks (polling): Your Server ──── "Any messages?" ────► Meta API Your Server ◄─── "No" ────────────── Meta API Your Server ──── "Any messages?" ────► Meta API (every few seconds) Your Server ◄─── "No" ────────────── Meta API Your Server ──── "Any messages?" ────► Meta API Your Server ◄─── "Yes, here it is" ── Meta API With webhooks (event-driven): User sends WhatsApp message │ ▼ Meta API ──── POST /webhook ──────────► Your Server { message data } │ ▼ Your server handles it instantly

This is why webhooks are fundamental to the WhatsApp Cloud API — there is no other way to receive incoming messages. You must expose a publicly accessible HTTPS endpoint and register it with Meta.

The two requests Meta sends to your webhook

RequestWhenPurpose
GET Once, when you first register or update the webhook URL in Meta Verification — Meta checks that the URL actually belongs to you and your server is reachable
POST Every time a relevant event happens (message received, message status update, etc.) Delivery — Meta sends you the event data

The three settings you configure

1

Callback URL

This is the full HTTPS URL of your webhook endpoint — the address Meta will POST messages to. For example:

URL
https://your-server.com/webhook

Requirements:

  • Must be HTTPS — plain HTTP is not accepted
  • Must be publicly accessible — Meta's servers need to reach it from the internet, so localhost or 127.0.0.1 will never work
  • Must respond to a GET request during verification (covered below)
  • Must respond to POST requests with HTTP 200 within 5 seconds
Where to host your server for free

Your webhook server needs a permanent, public HTTPS URL. The easiest approach is to deploy it to a free hosting service — they all support connecting your GitHub repository so every push auto-deploys, or you can deploy via Docker.

  • Render — connect your GitHub repo, choose "Web Service", free tier included. Your URL will be https://your-app.onrender.com
  • Railway — same GitHub-connect workflow, very fast deploys, generous free tier
  • Google Cloud Run — deploy from GitHub or a Docker container, scales to zero (free when idle), very reliable

All three give you a stable HTTPS URL that never changes — unlike temporary tunnelling tools where you'd have to re-register the webhook every time you restart your machine.

2

Verify Token (the secret)

The verify token is a string you make up yourself — any random text you choose. You set it in two places:

  1. In the Meta webhook configuration form (so Meta knows what to send)
  2. In your server's environment variables as VERIFY_TOKEN (so your server knows what to expect)

When you save the webhook URL in Meta, Meta immediately sends a GET request to your callback URL with three query parameters:

GET request from Meta
GET /webhook?hub.mode=subscribe&hub.verify_token=YOUR_VERIFY_TOKEN&hub.challenge=1158201444

Your server must:

  1. Check that hub.mode equals "subscribe"
  2. Check that hub.verify_token matches your stored VERIFY_TOKEN
  3. If both match — respond with HTTP 200 and the value of hub.challenge as the body
  4. If they don't match — respond with HTTP 403
The verify token is not a security token

The verify token is only used during the one-time URL verification handshake. It is not sent with every message — don't confuse it with your WHATSAPP_TOKEN (the access token). They are completely different things. Any string works as a verify token: "mySecretString", "abc123", etc.

3

Webhook fields — what to subscribe to

After the URL is verified, Meta shows you a list of webhook fields — the categories of events you want to receive. You need to subscribe to the right ones or events won't arrive.

FieldWhat it deliversSubscribe?
messagesIncoming messages from users, message status updates (sent, delivered, read), reactions✅ Yes — required
message_template_status_updateNotifications when a template is approved or rejectedOptional
account_updateChanges to your WhatsApp account (phone number quality, restrictions)Optional
business_capability_updateUpdates to your messaging tierOptional
messages field must be subscribed

Even after your webhook URL is verified, if the messages field is not subscribed, Meta will not forward any incoming messages to your server. The webhook will show a green checkmark but nothing will arrive. Always check this field is toggled on after verification.

How to configure the webhook in Meta — step by step

1

Deploy your server first

Your server must be running and publicly accessible before you can complete webhook setup. Deploy to a free hosting platform — Render, Railway, or Google Cloud Run all work well and connect directly to a GitHub repo. Copy the HTTPS URL your platform assigns — you'll need it in the next step.

2

Open webhook configuration

  1. Go to developers.facebook.com/apps and open your app
  2. In the left sidebar, click WhatsApp → Configuration
  3. Scroll to the Webhook section and click Edit
3

Fill in the two fields

FieldWhat to enterExample
Callback URL Your server's full HTTPS URL + /webhook https://your-app.onrender.com/webhook
Verify token Any string you choose — must match your VERIFY_TOKEN env var exactly mySecretVerifyToken

Click Verify and Save. Meta immediately sends the GET verification request to your server. If your server handles it correctly, you'll see a green checkmark and the configuration saves.

4

Subscribe to the messages field

After verification, you'll see the Webhook fields section appear. Find messages in the list and toggle it to Subscribed. Click Done.

Your webhook is now fully configured. Every message sent to your WhatsApp number will be forwarded to your server as a POST request to /webhook.

What a webhook POST payload looks like

When a user sends "Hello" to your WhatsApp number, Meta sends your server a POST request with a JSON body like this:

JSON (incoming message payload)
{
  "object": "whatsapp_business_account",
  "entry": [
    {
      "id": "YOUR_WABA_ID",
      "changes": [
        {
          "value": {
            "messaging_product": "whatsapp",
            "metadata": {
              "display_phone_number": "263775550000",
              "phone_number_id": "YOUR_PHONE_NUMBER_ID"
            },
            "contacts": [
              {
                "profile": { "name": "John Doe" },
                "wa_id": "263776001234"
              }
            ],
            "messages": [
              {
                "from": "263776001234",
                "id": "wamid.abc123...",
                "timestamp": "1700000000",
                "type": "text",
                "text": { "body": "Hello" }
              }
            ]
          },
          "field": "messages"
        }
      ]
    }
  ]
}

Key fields to extract from this payload:

PathValueWhat it is
entry[0].changes[0].value.messages[0].from"263776001234"Sender's phone number
entry[0].changes[0].value.messages[0].text.body"Hello"The message text
entry[0].changes[0].value.messages[0].type"text"Message type (text, image, interactive, etc.)
entry[0].changes[0].value.metadata.phone_number_idYour Phone Number IDWhich of your numbers received the message
Not every POST is a message

Meta also sends POST requests to your webhook for message status updates (sent → delivered → read) and other events. These payloads look similar but have no messages array — they have a statuses array instead. Always check that value.messages exists before trying to parse it, or your server will crash on status updates.

Common webhook problems and fixes

ProblemCauseFix
Verification fails — Meta shows an error Your server isn't running, URL is wrong, or the GET handler isn't returning the challenge correctly Test the URL in a browser first — it should show something. Check your GET handler returns hub.challenge as plain text
Webhook verified but no messages arrive messages field not subscribed, or WABA not subscribed to the app Check the webhook fields toggle; run the subscribed_apps curl command from Section 2
Duplicate messages delivered Your server took more than 5 seconds to respond with 200, so Meta retried Send res.sendStatus(200) immediately at the top of your POST handler, before any processing
Works in staging/dev but not after deploying to production Webhook URL in Meta still points to the old server URL, or env vars weren't set on the new hosting platform Re-register the webhook with the new server URL; set VERIFY_TOKEN and WHATSAPP_TOKEN in the platform's environment variable settings
Verification keeps failing after redeployment The VERIFY_TOKEN env var in Meta doesn't match the one in your deployed server They must be character-for-character identical — check for trailing spaces or mismatched casing
